# Diagnostic vs Structural Layer Note

## Purpose of this document

This note hardens one of the most important internal separations in the **Quantum–Spacetime Bridge** project:

> the distinction between **structural layers** and **diagnostic layers**

Its purpose is to prevent conceptual drift.

The project has now accumulated:
- multiple experiment blocks
- arrangement-sensitive scores
- shell-sensitive scores
- marker/carrier language
- shell-order language
- summary labels
- block-level decisions
- layered interpretations

All of these are useful.
But they are **not the same kind of thing**.

This note is a machine-room clarification that makes the separation explicit and strict.

The central question is:

> Which parts of the project are candidate structure, and which parts are only ways of reading, testing, or summarizing that structure?

This is an internal architecture note, not a public text.

---

## 1. Why this distinction must now be hardened

At an early stage of a project, it is normal to let:
- measurements
- interpretations
- and candidate structure

sit close together.

At the current stage, that is no longer safe.

The recent bridge work has already produced:

- primitive-structure hypotheses
- primitive-weighting and orderability notes
- shell as early derived order
- BMC and BMC-SX probe lines
- arrangement-sensitive metrics
- shell-sensitive metrics
- decision labels such as `undecided`, `carrier_leaning`, `shell_order_leaning`

If these are not sharply separated, the project risks a serious category error:

- diagnostic outcomes begin to sound like ontology
- readout surfaces begin to sound like primitive structure
- scaffold labels begin to sound like theory claims

This note is meant to prevent exactly that.

---

## 2. Minimal working distinction

The sharpest current separation is:

### Structural layer
A structural layer is part of the architecture that the theory is trying to describe as real, constitutive, derived, or organizing.

### Diagnostic layer
A diagnostic layer is a methodological layer used to:
- detect
- summarize
- classify
- compare
- or interpret

the structural layer.

A useful shorthand is:

> Structure is what the framework is about.  
> Diagnostics are how we currently inspect it.

This should remain a hard rule.

## 3. What belongs to the structural side

The following currently belong on the **structural** side of the project.

### 3.1 Distinguishable relational supports / endpoints
These are candidates for low-level support structure.

### 3.2 Weighted pairwise relation
This is currently the strongest low-level constitutive candidate.

### 3.3 Deep orderability / stratification capacity
This remains partly open, but is being treated as structurally serious.

### 3.4 Shell order
Current best placement:
- early derived
- but real
- and structurally robust

### 3.5 Local-group organization
Not primitive, but likely a real organizational layer rather than just a measurement artifact.

### 3.6 Bridge-facing scaffold behavior
This is higher-level and still provisional, but it belongs on the structural side more than on the diagnostic side.

The key point:
these are candidate features of the architecture itself.

---

## 4. What belongs to the diagnostic side

The following belong on the **diagnostic** side.

### 4.1 Readout scores
Examples:
- `arrangement_signal_score`
- `shell_boundary_disruption_score`
- `shell_crossing_fraction`
- `endpoint_load_shift_score`
- `pair_neighborhood_consistency_shift_score`

These are not the structure itself.
They are measurements or summaries of how the structure behaves under a particular probe design.

### 4.2 Block-level decision labels
Examples:
- `marker_supported`
- `carrier_leaning`
- `shell_order_leaning`
- `undecided`
- `test_not_informative`

These are diagnostic classifications.
They are not theory primitives and not structural entities.

### 4.3 Probe-family outcome language
Examples:
- preserving vs crossing
- buffered vs disrupted
- weakly_readable vs arrangement_sensitive

These are interpretive output terms tied to a given testing framework.

### 4.4 Summary bundles and report tables
Examples:
- `decision_summary.json`
- `summary.json`
- matrix or ladder readouts
- block-level markdown summaries

These are packaging layers for inspection, not structural layers of the theory.

---

## 5. Why the diagnostic layer is still very important

The fact that something is diagnostic does **not** make it unimportant.

This must be stated clearly.

Diagnostics are essential because they allow us to:

- detect structural differences
- separate informative from uninformative runs
- compare intervention families
- decide where the current architecture is sensitive
- avoid handwaving
- discipline interpretation

So the correct position is **not**:

- structural = real
- diagnostic = irrelevant

The correct position is:

- structural = candidate architecture
- diagnostic = disciplined access to that architecture

That is a much better and more scientifically honest distinction.

---

## 6. The main category error to avoid

The project must now avoid the following slide:

> “The score is high, therefore the score is the structure.”

That is wrong.

For example:

- a high `arrangement_signal_score` means the current diagnostic readout detects stronger structure-sensitive perturbation response
- it does **not** mean that `arrangement_signal_score` is itself part of the theory’s primitive or derived ontology

Likewise:

- a `shell_order_leaning` label means the current decision logic sees evidence compatible with shell as a significant order layer
- it does **not** mean that the label itself is a structural ingredient

This distinction is absolutely central now.

## 8. A subtle but important point

Some things sit near the boundary and must be handled carefully.

### Example: bridge-facing scaffold behavior

This is not purely diagnostic, because the project is genuinely trying to describe an emergent bridge-carrying organizational layer.

But our **current access** to it is mostly diagnostic.

So a boundary rule is useful:

> A structural candidate may be read through diagnostics, but that does not collapse the candidate into the diagnostic layer.

This applies especially to:
- scaffold behavior
- bridge relevance
- order preservation
- carrier suspicion

These are structurally intended notions, but presently accessed through provisional diagnostic instruments.

That is a delicate but necessary distinction.

---

## 9. What the recent experiments imply about this separation

The recent bridge and shell work makes the separation sharper, not weaker.

### 9.1 Experiments support structural depth
The shell ladder and weighted perturbation results strongly suggest that:
- weighting is constitutive
- shell is real derived order
- orderability is deep

These are structural claims.

### 9.2 Experiments refine diagnostics
At the same time, the readout-upgrade process showed that:
- diagnostics can be too weak
- diagnostics can be blind
- diagnostics can improve
- diagnostics can overread if not constrained

That is a major lesson.

So the project has already learned two things at once:

> structure is real  
> diagnostics are revisable

This is exactly why they must not be confused.

---

## 10. Weakest defensible current reading

The weakest defensible current reading is:

> The project now contains both structural hypotheses and diagnostic machinery, and the latter must remain explicitly subordinate to the former.

This is already stronger and cleaner than the earlier mixed language.

---

## 11. Strongest current internal reading worth adopting

The strongest current machine-room reading worth adopting is:

> The theory lives in the structural layers; the current scores, labels, and summaries are disciplined probes of those layers, not the layers themselves.

This is probably the single best current sentence to stabilize the whole architecture.

A shorter version is:

> Scores are probes, not ontology.

That one is especially useful.

---

## 12. What this means for writing and interpretation

The separation has direct practical consequences.

### 12.1 When writing structural notes
Use language such as:
- constitutive
- derived
- organizing
- stabilizing
- order-bearing
- scaffold-like

### 12.2 When writing diagnostic notes
Use language such as:
- detects
- indicates
- supports
- leans toward
- current readout suggests
- current probe is compatible with

### 12.3 What to avoid
Avoid silently sliding from:
- “score X rises”
to
- “therefore structure Y is proven”

That transition must always be argued, never smuggled.

---

## 13. Strategic consequence

This distinction protects the project in two important ways.

### 13.1 It protects the theory core
The structural architecture does not get blurred into its current test harness.

### 13.2 It protects scientific caution
The project can keep building strong diagnostics without pretending those diagnostics are already the thing itself.

That is a major maturity step.

---

## 14. Suggested operational rule for future work

A very useful machine-room rule from now on would be:

### Structural questions
Ask:
- what exists or is organized in the architecture?
- what is primitive?
- what is derived?
- what stabilizes what?

### Diagnostic questions
Ask:
- how do we see it?
- how do we compare it?
- how do we summarize it?
- how strong is the current support?

If a sentence mixes both, rewrite it.

That is a good internal discipline rule.
