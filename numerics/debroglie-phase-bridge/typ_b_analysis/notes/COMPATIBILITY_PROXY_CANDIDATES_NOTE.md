# COMPATIBILITY_PROXY_CANDIDATES_NOTE

## Proxy candidates for the missing compatibility layer

## 1. Purpose
This note continues the current bridge-mechanism refinement by asking a more operational question:

> **If the compatibility layer is real, where might we expect indirect traces of it in the current project observables?**

The project now distinguishes conceptually between:
- overlap
- compatibility
- binding / stabilization
- visible transition form

At present, however, the project does not yet possess a direct measured variable explicitly labeled “compatibility.”

The goal of this note is therefore:
- to identify plausible **proxy candidates**
- to separate proxies from downstream effects
- and to avoid prematurely equating existing observables with compatibility itself

---

## 2. Why this note is needed
The compatibility layer became necessary because the project now has a more refined mechanistic chain:

- overlap field
- compatibility layer
- bindable overlap
- bound transition mode
- bound transition structure

But current observables were not originally designed around this separation.  
This creates a risk:

- existing variables may be mistaken for compatibility itself,
- when they may instead reflect later stages such as stabilization or morphology.

So the task is not yet:
- “identify the compatibility metric”

but rather:
- “identify which current observables may carry indirect traces of compatibility.”

---

## 3. First distinction: proxy vs identity
A proxy candidate for compatibility should not automatically be treated as compatibility itself.

Three roles must be distinguished:

### 3.1 Upstream condition
A quantity that may reflect whether overlap is structurally fit to support a shared mode.

### 3.2 Stabilization effect
A quantity that becomes large only once a compatible mode has already stabilized.

### 3.3 Morphological effect
A quantity that becomes visible only once the stabilized mode has taken on organized structure.

This distinction is crucial.  
Otherwise the project risks collapsing:
- selection
- persistence
- and structure
into one layer.

---

## 4. Current main candidates and their likely roles

## 4.1 `simple_rigidity_surrogate`
### Current evidence
In the bridge-carrier sequence, `simple_rigidity_surrogate` behaves as:
- strongest internal anchor
- single-feature internal support carrier
- necessary for maintaining bridge-like structure

### Most plausible reading
This quantity most likely reflects:
- stabilization
- coherence persistence
- internal binding core

### Why it is probably not compatibility itself
Rigidity seems to answer:
- **What keeps the intermediate structure together once it exists?**

But compatibility should answer a slightly earlier question:
- **Can the overlap support a shared mode in the first place?**

So the current best reading is:

> `simple_rigidity_surrogate` is probably a proxy for the **successfully stabilized outcome** of compatibility, not for compatibility itself.

### Interim classification
- **strong downstream proxy**
- likely closer to **binding core** than to compatibility-layer identity

---

## 4.2 `grid_deviation_score`
### Current evidence
In the bridge-carrier sequence, `grid_deviation_score` behaves as:
- necessary co-carrier of fuller internal structure
- part of what makes the bridge structurally readable
- especially important for the fuller internal bridge morphology

### Most plausible reading
This quantity most likely reflects:
- organized transition form
- structural shaping
- visible morphology of the bridge

### Why it is probably not compatibility itself
Grid-related form seems to answer:
- **What organized shape does the stable intermediate mode take?**

But compatibility should again answer the earlier question:
- **Was a shared intermediate mode supportable before visible form emerged?**

So the current best reading is:

> `grid_deviation_score` is probably a proxy for **morphological organization after stabilization**, not for compatibility itself.

### Interim classification
- **strong morphology proxy**
- likely downstream of compatibility and stabilization

---

## 4.3 `distance_to_type_D`
### Current evidence
Current A–E results suggest:
- not an internal primary bridge carrier
- not a supported genuine H2 transfer stabilizer
- useful mainly as context/reference quantity

### Most plausible reading
This quantity likely reflects:
- embedding position
- structural reference location
- contextual relation to a type-D regime

### Why it is unlikely to be compatibility itself
Distance-like description may accompany compatibility, but does not by itself explain:
- shared supportability
- binding-capable overlap
- or coherent intermediate mode formation

So the current best reading is:

> `distance_to_type_D` is best treated as a **context / embedding variable**, not as a compatibility proxy of first rank.

### Interim classification
- **weak compatibility candidate**
- stronger as contextual descriptor than as compatibility marker

---

## 4.4 `spacing_cv`
### Current evidence
Current A–E results suggest:
- not an internal primary bridge carrier
- not a decisive transfer stabilizer
- useful as a context / regularity descriptor

### Most plausible reading
This quantity likely reflects:
- regularity or irregularity of local spacing structure
- corridor-like structural context
- not the core selection step itself

### Why it is unlikely to be compatibility itself
Spacing-like quantities may describe whether a region is regular, dispersed, corridor-like, or noisy.  
But compatibility should answer:
- whether an overlap can be jointly supported as a stable intermediate mode

This is a different role.

### Interim classification
- **weak compatibility candidate**
- likely contextual rather than selective

## 6. What might count as indirect compatibility traces
If compatibility is not directly measured, what indirect signs might still point to it?

## 6.1 Necessary-but-not-sufficient precursor behavior
A compatibility proxy might behave as:
- not identical with stabilization
- but necessary before stabilization becomes possible

This would mean:
- it correlates with success of bridge formation
- but not simply as a byproduct of already-formed structure

This kind of behavior has not yet been cleanly isolated.

---

## 6.2 Selective bridge appearance under same overlap availability
A compatibility trace might become visible when:
- overlap appears present in several cases,
- but only some cases stabilize into a readable bridge

Then compatibility would explain:
- why some overlap becomes bindable
- while some overlap remains only generic or residual

This may become one of the most important future discrimination tests.

---

## 6.3 Survival of shared mode under nontrivial mismatch
A compatibility-sensitive quantity might reveal whether two relational zones can still support a shared mode despite partial mismatch.

This would be analogous to:
- phase fit
- mode fit
- structural fit
- or donor-acceptor style state compatibility in chemistry analogies

No current variable is yet clearly isolated as this measure.

---

## 7. Current best hypothesis
The current best working hypothesis is:

> The project’s present observables do not yet directly measure compatibility. Instead, they appear to begin downstream of compatibility, with `simple_rigidity_surrogate` reflecting stabilization and `grid_deviation_score` reflecting organized structural form.

This means:
- the compatibility layer is likely **real conceptually**
- but still **missing operationally**

This is not a weakness of the idea.  
It is a clarification of where the project still lacks a dedicated variable.

---

but not yet:

---

## 9. Most important open question
The central unresolved question becomes:

> **What quantity, or combination of quantities, would distinguish “overlap that can bind” from “overlap that merely exists”?**

This is likely the sharpest next conceptual problem in the whole current bridge-mechanism line.

---

## 10. Minimal future test logic
A future test for compatibility proxies should ideally compare cases where:
- overlap or intermediate appearance is present in both cases
- but only one case stabilizes into a bridge-like structure

Then the task would be:
- identify which quantity or pattern separates the successful from the unsuccessful case

This would be closer to a true compatibility discrimination test than broad bridge/no-bridge summaries.

---

## 11. Strong internal synthesis sentence
A useful current internal sentence is:

> **The current bridge carriers likely describe what happens after compatibility has already succeeded: rigidity as stabilization, grid as morphology, distance/spacing as context. The actual compatibility layer itself is still missing as a distinct measured variable.**

A shorter version is:

> **Compatibility is currently a missing layer, not yet a measured carrier.**

Both are strong internal orientation sentences.

---

## 12. Bottom line
The current best internal reading is:

> The project’s present observables are best interpreted as downstream proxies rather than direct compatibility measures. `simple_rigidity_surrogate` most plausibly reflects stabilization, `grid_deviation_score` reflects organized bridge form, and `distance_to_type_D` plus `spacing_cv` describe context and embedding. The compatibility layer itself remains conceptually necessary but operationally still missing.

That is the present value of the proxy-candidate perspective.
