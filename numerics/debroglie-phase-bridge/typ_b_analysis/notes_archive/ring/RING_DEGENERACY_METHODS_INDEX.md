## 1. Purpose
This file is the entry point for the current ring degeneracy methods block.

Its purpose is to prevent the ring line from dissolving into a set of disconnected notes.  
Instead, it provides one place that explains:

- what the ring methods block is for,
- why it exists,
- which files belong to it,
- in what order they should be read,
- and how they fit into one practical workflow.

This is an index note, not a result note.

---

## 2. Why this methods block exists
The ring was identified as a special methodological case within the project.

It is not just another family.
It functions as a potential:
- resistant boundary case,
- symmetry-sensitive test family,
- and interpretive stress test for `delta_p` versus `delta_p2` readings.

Because of this, the ring needed tighter methodological discipline than an ordinary exploratory side-run.

The ring methods block therefore exists to ensure that any future ring degeneracy perturbation result is:
- provenance-clean,
- testable,
- interpretable,
- reviewable,
- and resistant to narrative inflation.

---

## 3. Core idea of the block
The central methodological idea is:

> **Do not allow an attractive ring result to enter the project as evidence unless goal, intervention, metrics, decision logic, reporting, and review are all already disciplined in advance.**

That is the reason this block exists.

## 5. What each file is for

### `RING_DEGENERACY_NUMBERS_PROVENANCE_CAUTION_NOTE.md`
Use this when attractive ring numbers appear without secure provenance.  
Function:
- quarantines uncertain numbers
- preserves hypothesis value without admitting them as evidence

### `RING_DEGENERACY_EVIDENCE_STANDARD_NOTE.md`
Defines what must be true for a ring result to count as a project-valid result.  
Function:
- evidence threshold
- status distinction between hypothesis, hint, and accepted result

### `RING_DEGENERACY_MINIMAL_PILOT_DESIGN_NOTE.md`
Defines the smallest clean pilot design.  
Function:
- turns ring speculation into a narrow, disciplined test question

### `RING_DEGENERACY_PRE_RUN_CHECKLIST.md`
Used before execution.  
Function:
- checks whether the run is ready to start
- prevents bad runs from starting methodically broken

### `RING_DEGENERACY_DECISION_LOGIC_NOTE.md`
Defines how outputs are to be interpreted.  
Function:
- prevents post hoc beautification
- enforces raw-metric-first reading

### `RING_DEGENERACY_RESULT_REPORT_TEMPLATE.md`
Defines how a finished run is to be written up.  
Function:
- standardizes reporting
- prevents inflated summaries

### `RING_DEGENERACY_OUTPUT_BUNDLE_SPEC.md`
Defines which files a valid run must leave behind.  
Function:
- ensures reproducibility and archive value

### `RING_DEGENERACY_RUN_FOLDER_CONVENTION_NOTE.md`
Defines how runs are named and stored.  
Function:
- preserves provenance and run hygiene

### `RING_DEGENERACY_POST_RUN_REVIEW_CHECKLIST.md`
Used after execution.  
Function:
- determines whether a run may remain draft, be reviewed, or needs rerun

## 7. Recommended practical workflow
The files are not just for reading; they define a workflow.

### Stage 1 — Before any run
Use:
- `RING_DEGENERACY_EVIDENCE_STANDARD_NOTE.md`
- `RING_DEGENERACY_MINIMAL_PILOT_DESIGN_NOTE.md`
- `RING_DEGENERACY_PRE_RUN_CHECKLIST.md`

Question:
- Is the pilot worth running, and is it methodically clean enough to start?

### Stage 2 — During run setup
Use:
- `RING_DEGENERACY_OUTPUT_BUNDLE_SPEC.md`
- `RING_DEGENERACY_RUN_FOLDER_CONVENTION_NOTE.md`

Question:
- Is the run being set up so that it will later be reproducible and archive-safe?

### Stage 3 — After run completion
Use:
- `RING_DEGENERACY_DECISION_LOGIC_NOTE.md`
- `RING_DEGENERACY_RESULT_REPORT_TEMPLATE.md`
- `RING_DEGENERACY_POST_RUN_REVIEW_CHECKLIST.md`

Question:
- What category does the result belong to, how should it be reported, and may it be promoted?

### Stage 4 — If unverified numbers appear
Use:
- `RING_DEGENERACY_NUMBERS_PROVENANCE_CAUTION_NOTE.md`

Question:
- Is this evidence, or only a hypothesis signal?

## 8. Internal philosophy of the block
The ring methods block follows a simple internal philosophy:

- a boundary case is not a playground for narrative excitement
- an attractive result is not automatically evidence
- raw metrics outrank labels
- provenance outranks enthusiasm
- review outranks momentum

This philosophy should remain visible whenever the ring is discussed.

---

## 9. What this block is trying to prevent
This methods block is specifically designed to prevent:

- provenance-uncertain tables becoming “results”
- winner-label flips being mistaken for robust physical transitions
- ring runs being reported without raw metric context
- pilot results being escalated into broad theoretical rescue claims
- weak or ambiguous runs being promoted because they are intellectually tempting

That is the negative function of the block.

---

## 10. What success would look like
This methods block has succeeded if, later, a reader can open the ring section and immediately see:

- which result was only a hypothesis
- which run was a clean pilot
- why a certain category was assigned
- what the limitations were
- and whether the result is synthesis-eligible or not

In other words:
success means the ring can be read without confusion.
