# PRE_READABLE_TYPE_B_FIRST_PASS_DECISION_LOGIC_NOTE

## Decision logic for the first pre-readable Type-B pass

**Date:** 2026-04-07  
**Status:** internal decision-logic note

---

## 1. Purpose
This note defines the decision logic for the first pre-readable Type-B pass.

Its purpose is to answer one practical question:

> **Once the winning subset has been reviewed through A1 and the readability-side boundary, how exactly should the case be classified without narrative drift?**

This note is needed because the block now already has:
- a selection chain,
- a winning-subset block,
- a first case-review workflow,
- and a result template.

What it still needs is one compact rule for turning review observations into an explicit case status.

This is a decision-logic note, not a result note.

---

## 2. Why a decision logic is needed
Without explicit decision logic, the pre-readable block becomes vulnerable to three familiar problems:

- a weak A1 gets over-read because the case is conceptually attractive
- a crossed readability boundary gets softened into “still sort of pre-readable”
- ambiguous cases get promoted as if they were already positive

So this note exists to enforce one thing:

> **selection and review must end in a disciplined case status, not in a mood**

That is the purpose of this note.

---

## 3. Core decision principle
The first-pass decision principle is:

> **A case may receive a positive pre-readable label only when organization-side activity is materially present and readability-side articulation remains below the formed-readability threshold.**

A shorter version is:

> **No positive label without strong enough A1 and still-weak readability.**

That is the governing rule.

---

## 4. Inputs to the decision logic
The decision logic should use exactly the following review inputs:

- winning subset fixed
- A1 judgment
- anti-overlap reading of A1
- anti-readability-leakage reading of A1
- readability-boundary judgment
- threshold classification: weak or crossed

No extra rescue input should be introduced mid-decision.

---

## 5. The three allowed case outcomes
The first pass should use exactly three case outcomes:

### Outcome A — Plausible pre-readable case
Use only when:
- A1 is materially active
- A1 is more than overlap-only presence
- A1 is still plausibly organization-side rather than readability-side
- readability boundary is weak

### Outcome B — Ambiguous / borderline case
Use when:
- some organization-side signal exists,
but
- either A1 strength, anti-overlap separation, or readability-boundary status remains too unclear for a clean positive or negative decision

### Outcome C — Not pre-readable
Use when:
- A1 is weak,
or
- A1 is effectively overlap-only,
or
- readability boundary is crossed,
or
- the case is already too readability-side formed

No fourth category should be introduced.

---

## 6. Positive decision rule
Assign:

## Plausible pre-readable case
only if all of the following hold:

1. **A1 materially active**
2. **A1 not reducible to overlap-only**
3. **A1 still plausibly on the pre-readable organization side**
4. **readability boundary = weak**
5. **no strong reason to treat the subset as already readability-side formed**

This should be treated as a strict conjunction.

If one of these fails clearly, do not assign the positive label.

---

## 7. Negative decision rule
Assign:

## Not pre-readable
if any of the following clearly holds:

1. **A1 weak**
2. **A1 effectively overlap-only**
3. **A1 already too readability-side in character**
4. **readability boundary crossed**
5. **subset already clearly readability-side formed**

This should also be treated strictly.

A case that clearly satisfies one of these negative conditions should not be upgraded through narrative rescue.

---

## 8. Borderline decision rule
Assign:

## Ambiguous / borderline case
only when neither the positive nor the negative rule can be applied cleanly.

Typical reasons:
- A1 is nontrivial but not clearly strong enough
- overlap-vs-organization distinction remains uncertain
- readability boundary sits too close to threshold to classify cleanly
- subset definition still introduces too much interpretive uncertainty

This category is necessary, but it must remain narrow.

It is not a soft “probably positive.”
It means:
- not yet decided cleanly.

---

## 9. Recommended decision sequence
The case status should be assigned in this order:

### Step 1
Check the negative kill conditions first:
- readability boundary crossed?
- A1 weak?
- A1 overlap-only?

If yes, assign **Not pre-readable** unless there is a clear reason this is genuinely borderline.

### Step 2
If no kill condition applies, check the full positive conjunction:
- A1 materially active?
- more than overlap?
- still organization-side?
- readability weak?

If yes, assign **Plausible pre-readable case**.

### Step 3
If neither a clean positive nor a clean negative applies, assign **Ambiguous / borderline case**.

This order reduces inflation risk.

---

## 10. What the decision logic must not do
The decision logic must not:

- infer positivity from weak readability alone
- infer positivity from conceptual elegance
- treat “almost positive” as positive
- keep a case in limbo without assigning one of the three statuses
- use broader ontology to rescue an unclear first-pass case

These are invalid decision habits.

---

## 11. Safe summary sentence
A useful internal sentence is:

> The first pre-readable Type-B case may be labeled positive only if A1 is materially active as organization-side structure while readability remains below threshold; otherwise the case is either ambiguous or not pre-readable.

A shorter version is:

> **Strong enough A1 plus still-weak readability, or no positive label.**

Both are good steering lines.

---

## 12. Bottom line
The first pre-readable Type-B pass should end in exactly one of three statuses:

- **Plausible pre-readable case**
- **Ambiguous / borderline case**
- **Not pre-readable**

The positive label should be granted only under the full conjunction:

- A1 materially active
- not overlap-only
- still organization-side
- readability boundary weak

That is the current decision-logic standard for the first pre-readable Type-B pass.
