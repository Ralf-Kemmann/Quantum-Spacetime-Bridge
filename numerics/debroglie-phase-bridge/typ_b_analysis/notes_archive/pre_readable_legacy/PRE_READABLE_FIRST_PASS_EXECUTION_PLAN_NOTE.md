# PRE_READABLE_FIRST_PASS_EXECUTION_PLAN_NOTE

## Execution plan for the first pre-readable comparison pass

**Date:** 2026-04-07  
**Status:** internal execution-planning note

---

## 1. Purpose
This note turns the current pre-readable methods block into one concrete first-pass execution plan.

The project now already has:

- a methods index
- a current-status note
- candidate marker families
- Candidate Marker A1
- a readability-side boundary
- a readability threshold rule
- a first comparison rule
- an executive note
- and a first-pass result template

What is still needed is one practical bridge from method to action:

> **What exactly has to be fixed and executed so that the first pre-readable pass can actually begin without improvisation?**

This note provides that bridge.

---

## 2. Scope of the first pass
The first pass remains intentionally narrow.

It is **not** meant to:
- prove pre-readable organization in general
- close the organization/readability distinction
- support a broad morphology claim
- or justify any larger ontology

Its only purpose is:

> **to determine whether at least one case can be shown to satisfy the current A1-versus-readability-boundary rule strongly enough to count as a plausible pre-readable case.**

That is the whole scope.

---

## 3. Fixed first-pass comparison structure
The first pass should use exactly this structure:

- **Organization-side marker:** A1 = structured support coherence
- **Readability-side boundary marker:** `grid_deviation_score`
- **Readability threshold rule:** weak vs crossed
- **Comparison rule:** plausible pre-readable only if A1 is materially active while readability remains weak

No second organization-side marker should be added in this first pass.

Reason:
- the first pass should test the cleanest first marker/boundary pair only
- and remain narrow enough that failure stays interpretable

---

## 4. What must be fixed before execution
Before the first pre-readable pass starts, the following items must be fixed explicitly.

## 4.1 A1 operational shell
For A1, define:
- the exact neighborhood definition
- the exact support-coherence term
- the exact fragmentation/conflict term
- the exact aggregation rule
- the earliest safe computation stage

This is required because A1 is currently:
- conceptually strong
- but still formula-soft

---

## 4.2 Readability-side threshold application
For the readability boundary, define:
- the exact threshold condition applied to `grid_deviation_score`
- how “weak” is assigned
- how “crossed” is assigned
- and how threshold consistency is maintained across cases

This is required because the boundary marker exists conceptually, but still needs one explicit pass-level application rule.

---

## 4.3 Case family or case window
At least one case family or case window must be fixed.

This should answer:
- where the first pre-readable pass will actually look

It should not remain abstract.
The first pass must be able to say:
- which cases were tested
- and why those cases were chosen

---

## 4.4 Overlap-side control awareness
Even though this block is not the compatibility block, a minimal overlap-awareness check should still be kept in mind:

- A1 must not be accepted as early organization if it is only overlap-only presence in disguise

So the first pass should include at least a minimal anti-overlap check in the case review.

This does not need to become a full baseline block yet, but it must remain visible.

---

## 5. First-pass execution sequence
Once the above pieces are fixed, the pre-readable first pass should proceed in exactly this order.

### Step 1 — Finalize A1 definition for the pass
Lock the first-pass form of A1.

No conceptual drift after this point.

---

### Step 2 — Lock the readability threshold rule
Lock:
- the readability boundary marker
- the threshold condition
- and the weak/crossed assignment logic

No mid-pass threshold changes.

---

### Step 3 — Fix the case or case window
Choose:
- one case
or
- one narrow case family / window

Do not widen case selection mid-pass.

---

### Step 4 — Evaluate A1 for the chosen case
Ask:
1. is A1 materially active?
2. is it more than overlap-only presence?
3. is it still plausibly pre-readable rather than already readability-side?

Record result immediately.

---

### Step 5 — Evaluate the readability boundary for the same case
Ask:
1. is `grid_deviation_score` still below the formed-articulation threshold?
2. should the boundary be classified as weak or crossed?

Record result immediately.

---

### Step 6 — Apply the comparison rule
Assign one of the following outcomes:

- **Plausible pre-readable case**
- **Not pre-readable**
- **Ambiguous / borderline case**

Do not invent a fourth category.

---

### Step 7 — Record promotion decision
State explicitly:
- no promotion
- keep as exploratory-only case
- carry forward as plausible pre-readable candidate case
- or revisit only after marker/boundary refinement

This decision must be written down, not implied.

---

## 6. First-pass stopping discipline
The first pass must include explicit stopping discipline.

### 6.1 No marker expansion mid-pass
Do not add:
- asymmetry markers
- persistence-like markers
- or second-wave marker families

If A1 fails, that is a legitimate first-pass result.

---

### 6.2 No threshold shifting mid-pass
Do not redefine the readability threshold because a case almost qualified.

---

### 6.3 No narrative rescue
Do not upgrade a weak or ambiguous case to “basically pre-readable” without an explicit methodological reason.

Aesthetic attraction is not enough.

---

## 7. Minimal artifacts the first pass should produce
The first pass should minimally leave behind:

- A1 marker definition note
- readability boundary note
- readability threshold rule note
- first-pass case result record
- one short promotion-status statement

This is the minimum execution bundle.

---

## 8. Acceptable first-pass end states
The first pass should be considered complete if it ends in one of these states.

### End state 1 — No plausible pre-readable case found
Interpretation:
- the current marker/boundary pair did not yet isolate a usable case

### End state 2 — Ambiguous / borderline case found
Interpretation:
- the distinction may be present, but marker or threshold refinement is still needed

### End state 3 — One plausible pre-readable case found
Interpretation:
- one case deserves carry-forward as the current best candidate pre-readable example

All three are acceptable as long as they are recorded cleanly.

---

## 9. What must not be claimed afterwards
After the first pass, the project must still not claim:

- pre-readable organization proven in general
- readability separation closed
- early structure experimentally established in full
- or broad bridge theory support

The strongest acceptable statement is something like:

> One case has survived first-pass comparison strongly enough to remain under consideration as a plausible pre-readable case.

That is the upper limit for now.

---

## 10. Safe internal summary sentence
A good internal steering sentence is:

> The first pre-readable pass begins only after A1, the readability threshold rule, and the case window are all fixed; after that, the pass should run narrowly, stop cleanly, and leave an explicit case-status decision behind.

A shorter version is:

> **Fix A1, lock the readability threshold, choose the case, run the pass, record the status, stop.**

Both are good current steering lines.

---

## 11. Bottom line
The pre-readable first pass is now methodically close to executable.

What remains to be fixed before it can begin is:

- A1 operational shell
- readability threshold application
- one first case or case window

After that, the first pass should be run narrowly and finished cleanly.

That is the current execution plan.
