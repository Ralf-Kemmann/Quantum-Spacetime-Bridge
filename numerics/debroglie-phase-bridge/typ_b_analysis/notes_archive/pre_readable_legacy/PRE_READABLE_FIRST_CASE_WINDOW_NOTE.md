# PRE_READABLE_FIRST_CASE_WINDOW_NOTE

## First concrete case type / case window for the pre-readable first pass

**Date:** 2026-04-07  
**Status:** internal case-window note

---

## 1. Purpose
This note defines the first concrete case type, or case window, for the pre-readable first pass.

Its purpose is to answer the next narrowing question:

> **Where should the first pre-readable pass actually look if it is to test whether early organization can be active before readable articulation becomes strong enough to count as formed?**

This is needed because the pre-readable block is now methodically prepared, but still needs one explicit observation window.

This is a case-window note, not a result note.

---

## 2. Why a concrete case window is needed
The pre-readable first pass can no longer remain purely architectural.

It already has:
- one organization-side marker: **A1**
- one readability-side boundary: **`grid_deviation_score`**
- one threshold rule
- one comparison rule
- one result template
- and one execution plan

What is still missing is the place where the first pass should actually look.

Without that, the block remains structurally complete but operationally floating.

So the first case window must now be fixed explicitly.

---

## 3. Selection principle for the first case window
The first case window should be chosen by one principle above all:

> **Look where organization may plausibly already be emerging, but where readable form is still not expected to be maximally strong.**

That means the first case window should avoid both extremes:

### Too early
Cases where essentially nothing has organized yet.

### Too late
Cases where readable structure is already clearly formed and the pre-readable question has mostly passed.

The best first window is therefore an intermediate one:
- structured enough to make A1 meaningful,
- but still early enough that the readability boundary may remain weak.

---

## 4. Recommended first case-window type
### Proposed first case-window type
Use a:

> **transitional / intermediate organization window in which support-side structure is already nontrivial but readability-side articulation is not yet clearly formed**

This is the recommended first case-window type.

A shorter version is:

> **structured-but-not-yet-readable window**

This should be the guiding case logic.

---

## 5. Why the first case should be transitional
A transitional window is the best first target because the pre-readable hypothesis is inherently about layered timing.

If the case is too late:
- the readability boundary will likely already be crossed

If the case is too early:
- A1 may remain too weak or overlap-like to be useful

So the ideal first-pass case is one where:
- early organization has a realistic chance to be active
- while the readability-side boundary still has a realistic chance to remain below threshold

That is exactly why the first case should be transitional.

---

## 6. Operational case-window definition
The first case window should therefore be defined as:

> **A local or mesoscopic window near emerging bridge-support organization, but before clearly readable articulation is already established.**

This is the current operational case-window definition.

It is narrow enough to act on, but still general enough to be implemented across an appropriate case family.

---

## 7. What characteristics the first case window should have
The chosen first window should ideally satisfy most of the following:

- nontrivial local relational structure already present
- some reason to expect structured support coherence
- not yet strongly readability-side formed
- not already obviously morphology-dominated
- not fully stabilized in a way that erases the pre-readable question
- interpretable enough that A1 and the readability boundary can both be evaluated meaningfully

This is the right first-pass profile.

---

## 8. What kinds of cases should currently be avoided
The first pre-readable pass should avoid the following case types at the start:

### 8.1 Fully formed readability-side cases
Cases where readable bridge-form is already strong and obvious.

Why avoid:
- they are too late
- the boundary is likely already crossed

### 8.2 Barely organized weak-noise cases
Cases where almost no structured organization is plausibly active yet.

Why avoid:
- A1 is likely too weak
- the pass then teaches very little

### 8.3 Strongly special / exotic edge cases
Cases whose specialness overwhelms the basic layered comparison.

Why avoid:
- they may be interesting later
- but are poor first-pass calibration targets

The first pass should start with a case that is hard enough to matter but not so special that everything becomes ambiguous.

---

## 9. Recommended first-pass case-window language
A safe current wording is:

> The first pre-readable pass should begin with a transitional local or mesoscopic case window in which support-side organization is already plausibly emerging, while readability-side articulation is still not clearly formed.

A shorter version is:

> **Choose the first window where organization may already be there, but readability is not yet clearly there.**

This is the right internal steering language.

---

## 10. Relationship to A1
The reason this case window is chosen is that A1 needs a fair chance to become active.

If the window is too late:
- A1 may be active, but no longer meaningfully pre-readable

If the window is too early:
- A1 may not yet separate itself from overlap-only presence

So the first case window must sit where A1 has a realistic chance to show:
- structured early organization
without
- already standing inside clearly formed readability

That is why case-window choice matters so much.

---

## 11. Relationship to the readability boundary
The first case window is also chosen to give the readability boundary a meaningful job.

The boundary must have a realistic chance to remain:
- **weak**

If the window is already too readability-side late, then the pass becomes trivial:
- boundary crossed
- not pre-readable

So the first case should live in the narrow zone where:
- the boundary might still not yet be crossed

That is the point of the first pass.

---

## 12. Recommended practical implementation choice
For the first concrete implementation, the case window should be chosen from:

> **a transitional segment / subset / local regime slice of the existing Type-B bridge-support landscape where support-side organization is already visible enough to be nontrivial, but later form articulation is not yet strongly established**

This is the best current practical formulation.

It intentionally points toward:
- transitional internal windows
rather than
- global final states

That is methodically correct.

---

## 13. First-pass acceptance logic for the case window
A case window should be accepted for first-pass use only if it appears to satisfy this basic precondition:

- A1 could plausibly become active here
and
- the readability boundary could plausibly still remain weak here

If one of those is already obviously false, the window is a poor first-pass choice.

This should be checked before running the pass.

---

## 14. Safe summary sentence
A useful internal sentence is:

> The first pre-readable pass should begin in a transitional window where structured support organization plausibly begins before readable articulation has clearly formed.

A shorter version is:

> **First case window: emerging organization, not yet formed readability.**

Both are good steering lines.

---

## 15. Bottom line
The first concrete case type / case window for the pre-readable pass should be:

> **a transitional local or mesoscopic bridge-support window in which organization is plausibly already emerging while readability-side articulation is still not clearly formed**

This is the correct first place to test the pre-readable comparison rule.

That is the current first case-window standard for the pre-readable block.
