# COMPATIBILITY_FIRST_PASS_FIXED_COMPARISON_FRAME_NOTE

## Fixed comparison frame for the first compatibility-candidate pass

**Date:** 2026-04-07  
**Status:** internal comparison-frame note

---

## 1. Purpose
This note fixes the complete comparison frame for the first compatibility-candidate pass.

Its purpose is to gather in one place the five structural components that now define the first pass:

- Candidate A
- Candidate B
- overlap-only baseline
- later-stabilization boundary
- downstream outcome proxy

This note is needed because the current methods block is now rich enough that the first pass should no longer be reconstructed from many separate notes alone.

Instead, the first pass should have one compact operational frame that states:

> **what is being compared, against what, and for what downstream purpose**

This is a frame note, not a result note.

---

## 2. Why this fixed frame is needed
The compatibility first pass is now methodically well prepared.

But without one fixed comparison frame, the risk remains that:
- the candidate pair is remembered loosely
- the lower baseline drifts
- the upper boundary is treated inconsistently
- the downstream target stays vague
- or different readers reconstruct different “first passes”

That would weaken the block.

So this note fixes the first-pass comparison frame explicitly.

---

## 3. Core first-pass question
The first pass should now be read as asking only this:

> **Can either Candidate A or Candidate B add meaningful pre-stabilization value beyond overlap-only, while remaining distinct from later stabilization and still relating to one narrow later bridge-support outcome?**

That is the complete first-pass question.

---

## 4. Fixed first-pass candidate pair

## 4.1 Candidate A
**Name:** local support coherence  
**Role:** coherence-side compatibility candidate

Candidate A asks:
- whether local relation environments already show internal supportability beyond mere overlap

Its intended mechanism position is:
- after overlap
- before stabilization
- before later form/readability

---

## 4.2 Candidate B
**Name:** early removal fragility  
**Role:** fragility-side compatibility candidate

Candidate B asks:
- whether local relations already reveal disproportionate support dependence under early weakening/removal

Its intended mechanism position is also:
- after overlap
- before stabilization
- before later form/readability

---

## 4.3 Why these two belong together
Candidate A and Candidate B are the correct first-pass pair because they probe the same missing layer from two complementary Tier-1 sides:

- A = local supportability by coherence
- B = local supportability by fragility

This gives the first pass:
- one constructive window
- one stress-response window

without candidate sprawl.

---

## 5. Fixed lower floor: overlap-only baseline
The lower comparison floor for the first pass is:

- **overlap-only baseline**

Its role is to represent:
- raw relational presence
- descriptive overlap magnitude
- local overlap availability

and nothing more.

It must not contain:
- local agreement logic
- conflict penalties
- removal sensitivity
- stabilization logic
- morphology/readability logic

So the lower-floor question is:

> **Can Candidate A or B do more than mere overlap?**

That is the lower-bound comparison.

---

## 6. Fixed upper ceiling: later-stabilization boundary
The upper comparison ceiling for the first pass is:

- **later-stabilization boundary**, with preferred current proxy:
- `simple_rigidity_surrogate`

Its role is not to serve as the goal of the candidates.
Its role is to define what is already too late to count as compatibility-side.

So the upper-boundary question is:

> **Does Candidate A or B remain earlier than achieved support / persistence, or has it already drifted into stabilization?**

That is the upper-bound comparison.

---

## 7. Fixed downstream target class
The first pass should use:

- **one narrow later bridge-support-related downstream outcome proxy**

This target should be:
- later than Candidate A and Candidate B
- narrower than full readability or morphology
- distinct from the overlap floor
- distinct from the rigidity-side boundary

It should function as:
- the first later outcome against which the candidates are judged

So the downstream question is:

> **Does Candidate A or B relate meaningfully to a narrow later support-related outcome without collapsing into late-stage success signals?**

That is the target-side comparison.

---

## 8. First-pass structure in one line
The full first-pass structure is therefore:

`Candidate A / Candidate B`
tested against
`overlap-only baseline`
bounded above by
`later-stabilization boundary`
and judged relative to
`one narrow downstream support-related outcome proxy`

This is the complete frame in one line.

---

## 9. Comparison logic inside the fixed frame
Within this fixed frame, each candidate should be processed by the same sequence:

### Step 1
Early enough?
- if no, reject

### Step 2
Better than overlap-only?
- if no, reject

### Step 3
Distinct from later stabilization?
- if no, reject or downgrade heavily

### Step 4
Interpretably related to the downstream outcome proxy?
- if no, keep at most exploratory

### Step 5
Assign candidate category:
- rejected candidate
- exploratory-only candidate
- weak but non-tautological candidate
- promising pre-stabilization candidate

This is the fixed decision chain.

That means the first pass is no longer just “ready in spirit.”

That is the main gain of this note.

That is already a strong step, but it is still not closure.
