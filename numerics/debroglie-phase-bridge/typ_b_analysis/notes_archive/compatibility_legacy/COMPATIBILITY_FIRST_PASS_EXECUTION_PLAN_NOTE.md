# COMPATIBILITY_FIRST_PASS_EXECUTION_PLAN_NOTE

## Execution plan for the first compatibility-candidate pass

**Date:** 2026-04-07  
**Status:** internal execution-planning note

---

## 1. Purpose
This note turns the current compatibility-methods block into one concrete first-pass execution plan.

The project now already has:

- a minimal compatibility pilot frame
- candidate evaluation logic
- a candidate priority structure
- a baseline-comparison standard
- a first test matrix
- a first-pass workflow
- a first-pass result template
- Candidate A and Candidate B as defined first-pass candidates
- and an explicit candidate-pair note

What is still needed is one practical bridge from method to action:

> **What exactly has to be fixed and executed so that the first compatibility pass can actually begin without improvisation?**

This note provides that bridge.

---

## 2. Scope of the first pass
The first pass remains intentionally narrow.

It is **not** meant to:
- solve the compatibility layer
- optimize many candidates
- prove a final mechanism
- or support a large synthesis claim

Its only purpose is:

> **to determine whether Candidate A or Candidate B survives disciplined first comparison against overlap-only and a later stabilization boundary.**

That is the whole scope.

---

## 3. Fixed first-pass candidate pair
The first pass should use exactly this pair:

- **Candidate A:** local support coherence
- **Candidate B:** early removal fragility

No third candidate should be added in this first pass.

Reason:
- small enough to stay disciplined
- broad enough to test two complementary Tier-1 windows
- narrow enough that rejection remains clean

---

## 4. What must be fixed before execution
Before the first pass starts, the following items must be fixed explicitly.

## 4.1 Candidate A operational shell
For Candidate A, define:
- the exact local neighborhood definition
- the exact agreement term
- the exact conflict term
- the exact aggregation rule
- the earliest safe computation stage

This is required because Candidate A is currently:
- conceptually strong
- but still formula-soft

---

## 4.2 Candidate B operational shell
For Candidate B, define:
- the exact removal or weakening operation
- the exact local structure to which removal is applied
- the exact support-response measure
- the earliest safe computation stage
- the guardrail against late-stage stabilization leakage

This is required because Candidate B is currently:
- conceptually plausible
- but timing-sensitive

---

## 4.3 Overlap-only baseline
One explicit overlap-only baseline must be fixed.

This should be:
- simple
- direct
- and weak enough to serve as a true lower baseline

It should not already contain:
- local coherence logic
- or hidden support structure

---

## 4.4 Later stabilization proxy
One explicit later stabilization boundary must be fixed.

Default candidate:
- `simple_rigidity_surrogate`

Reason:
- this is the clearest current downstream support/persistence anchor

This proxy defines the upper comparison boundary.

---

## 4.5 Optional trivial control
If possible, define one weak control proxy such as:
- smoothed local overlap magnitude
- simple local density
- or another deliberately low-information scalar

This is not mandatory, but highly useful.

---

## 4.6 Downstream outcome proxy
One narrow downstream outcome proxy must be fixed before the first pass.

It should be:
- later than the candidate signals
- narrow enough to remain interpretable
- and not so broad that every weak trend looks good

The first pass should not begin with a vague downstream target.

---

## 5. First-pass execution sequence
Once the above pieces are fixed, the first pass should proceed in exactly this order.

### Step 1 — Finalize candidate definitions
Lock the current first-pass forms of:
- Candidate A
- Candidate B

No conceptual drift after this point.

---

### Step 2 — Lock the comparison frame
Lock:
- overlap-only baseline
- later stabilization proxy
- optional trivial control
- downstream outcome proxy

No mid-pass baseline changes.

---

### Step 3 — Evaluate Candidate A first
Run Candidate A through the decision chain:

1. early enough?
2. better than overlap-only?
3. distinct from later stabilization?
4. interpretable?
5. optional trivial-control check
6. assign candidate category

Record result immediately.

---

### Step 4 — Evaluate Candidate B second
Run Candidate B through the same exact decision chain.

No separate special treatment.
No relaxed standard because Candidate B is more difficult.

Record result immediately.

---

### Step 5 — Assign first-pass global outcome
After A and B are both judged, assign exactly one global first-pass outcome:

- **Outcome A — No useful first-pass candidate**
- **Outcome B — Weak but non-tautological candidate found**
- **Outcome C — Promising pre-stabilization candidate found**

No fourth category.

---

### Step 6 — Record promotion decision
State explicitly:

- no candidate promoted
- one candidate promoted for second-pass refinement
- or both candidates promoted

This decision must be written down, not implied.

---

## 6. First-pass stopping discipline
The first pass must include explicit stopping discipline.

### 6.1 No candidate expansion mid-pass
Do not add:
- sign/sector contrast
- structured asymmetry
- or other rescue candidates

If A and B fail, that is a legitimate first-pass result.

---

### 6.2 No baseline shifting mid-pass
Do not redefine:
- overlap baseline
- stabilization boundary
- or downstream outcome proxy

because a candidate performed badly.

---

### 6.3 No narrative rescue
Do not reclassify a failed candidate as “basically promising” unless a specific methodological reason justifies a redefinition.
Aesthetic attraction is not enough.

---

## 7. Minimal artifacts the first pass should produce
The first pass should minimally leave behind:

- Candidate A definition note
- Candidate B definition note
- fixed overlap baseline note or explicit definition block
- fixed stabilization proxy declaration
- optional trivial-control definition
- first-pass result record
- one short summary sentence on promotion status

This is the minimum execution bundle.

---

## 8. Acceptable first-pass end states
The first pass should be considered complete if it ends in one of these states.

### End state 1 — Both rejected
Interpretation:
- first-pass candidate space did not yet yield a useful signal

### End state 2 — One weak but non-tautological survivor
Interpretation:
- one candidate deserves second-pass refinement, but not strong use yet

### End state 3 — One promising pre-stabilization survivor
Interpretation:
- one candidate deserves focused follow-up as the current best compatibility lead

### End state 4 — Both survive weakly
Interpretation:
- both candidates remain in play, but further narrowing is still required

All four are acceptable as long as they are recorded clearly under the allowed global-outcome logic.

---

## 9. What must not be claimed afterwards
After the first pass, the project must still not claim:

- compatibility found
- compatibility measured
- bridge selection solved
- mechanism closed

The strongest acceptable statement is something like:

> A first-pass candidate has survived disciplined comparison and deserves focused refinement.

That is the upper limit for now.

---

## 10. Safe internal summary sentence
A good internal steering sentence is:

> The first compatibility pass begins only after Candidate A, Candidate B, the overlap baseline, the rigidity boundary, and the downstream outcome proxy are all fixed; after that, the pass should run narrowly, stop cleanly, and leave an explicit promotion decision behind.

A shorter version is:

> **Fix the pair, lock the baselines, run the pass, record the promotion status, stop.**

Both are good current steering lines.

---

## 11. Bottom line
The compatibility first pass is now methodically close to executable.

What remains to be fixed before it can begin is:

- Candidate A operational shell
- Candidate B operational shell
- overlap-only baseline
- later stabilization proxy
- optional trivial control
- downstream outcome proxy

After that, the first pass should be run narrowly and finished cleanly.

That is the current execution plan.
