# COMPATIBILITY_FIRST_MINIMAL_RUN_NOTE

## Minimal-run note for the first compatibility pass

**Date:** 2026-04-07  
**Status:** internal minimal-run note

---

## 1. Purpose
This note defines the first **minimal run** for the compatibility first pass.

Its purpose is to answer one practical question:

> **How should the first compatibility run be executed in the smallest disciplined form that is still methodologically meaningful?**

This note is not a numerical run report.
It is the run-design note for the first deliberately minimal launch.

---

## 2. Why a minimal run is the correct next step
The compatibility block is now no longer missing structure.

It already has:

- accepted first local context
- Candidate A1 operationalization
- Candidate B1 operationalization
- overlap-only lower floor
- later-stabilization upper ceiling
- DLBSR-class downstream target
- first readout logic
- first qualitative content readout
- updated outcome language
- updated promotion language

That means the next correct move is **not**:
- more framework inflation

It is:
- one first disciplined run

The right first run should be:
- small
- hard-bounded
- and non-rescuing

That is why a minimal run is the correct next step.

---

## 3. Core principle
The first run should obey one central principle:

> **Run one accepted local context through the full A1/B1 comparison chain once, record the outcome, and stop before any adaptive redesign begins.**

A shorter version is:

> **One context, one pass, one record, then stop.**

That is the governing principle.

---

## 4. Scope of the minimal run
The first minimal run should use exactly:

- **one accepted local bridge-adjacent context**
- **one A1 readout**
- **one B1 readout**
- **one overlap-only lower-floor reading**
- **one rigidity-side ceiling reading**
- **one DLBSR-class downstream reading**
- **one first result record**

It should **not** yet use:

- multiple contexts
- multiple neighborhood redesigns
- multiple perturbation variants
- rescue reruns
- or exploratory branch expansion

This is the correct first-run scope.

---

## 5. Minimal-run sequence
The first minimal run should proceed in exactly this order.

### Step 1 — Lock the accepted local context
Use the already accepted first-pass local context.

Do not change:
- focal unit
- neighborhood boundary
- or launch frame

once the run begins.

---

### Step 2 — Read Candidate A1
Run the A1 readout through its three checks:

- internal activity
- anti-overlap separation
- anti-stabilization separation

Assign one status:
- materially active
- weak / ambiguous
- collapsed

Record the result immediately.

---

### Step 3 — Read Candidate B1
Run the B1 readout through its three checks:

- internal activity
- anti-overlap separation
- anti-stabilization separation

Assign one status:
- materially active
- weak / ambiguous
- collapsed

Record the result immediately.

---

### Step 4 — Apply the fixed frame
Interpret both readouts relative to:

- overlap-only baseline
- later-stabilization boundary
- DLBSR-class

Do not redefine any of these mid-run.

---

### Step 5 — Assign the global first-pass outcome
Use the fixed outcome vocabulary:

- Outcome A — No useful first-pass candidate
- Outcome B — Weak but non-tautological candidate found
- Outcome C — Promising pre-stabilization candidate found

Assign one outcome only.

---

### Step 6 — Assign the promotion language
Use the updated promotion logic.

Allowed states:
- no candidate promoted
- one candidate promoted for second-pass refinement
- both candidates promoted

Record one explicit decision only.

---

### Step 7 — Stop
After the first result record exists:

- do not change the context
- do not soften the boundary
- do not rescue a weak candidate
- do not run a second context in the same minimal-run note

The first minimal run ends here.

---

## 6. What the minimal run is trying to test
The first minimal run is not trying to prove the whole selector layer.

It is trying to test something much narrower:

> **Does the current qualitative asymmetry—A1 currently leading, B1 still in play but riskier—survive one disciplined first-pass execution in a fixed accepted context?**

That is the real question.

This makes the run scientifically useful even if the outcome is modest.

---

## 7. What the minimal run must not become
The minimal run must not become:

- a hidden exploratory campaign
- a context-tuning exercise
- a boundary-relaxation exercise
- a second-pass refinement in disguise
- or a proof note

It is only:
- the first strict first-pass launch

That must remain explicit.

---

## 8. Acceptable minimal-run end states
The first minimal run should be considered successful if it ends in one of the following disciplined states:

### End state 1 — A1 remains the weak but clearer lead
Interpretation:
- the current qualitative asymmetry survives first launch

### End state 2 — A1 and B1 both weaken
Interpretation:
- the present qualitative optimism was too generous

### End state 3 — B1 catches up or overtakes qualitatively
Interpretation:
- the present asymmetry needs revision

### End state 4 — Both remain in play without clear separation
Interpretation:
- the first launch is informative but inconclusive

All four are acceptable as long as they are recorded cleanly.

---

## 9. Safe summary sentence
A useful internal sentence is:

> The first compatibility minimal run should test the current A1-vs-B1 qualitative asymmetry under one fixed accepted context, one fixed frame, and one non-rescuing first-pass result record.

A shorter version is:

> **One fixed context, one A1/B1 pass, one outcome, then stop.**

Both are good steering lines.

---

## 10. Bottom line
The first compatibility minimal run should now be defined as:

- one accepted local context
- one A1 readout
- one B1 readout
- one lower/upper/downstream frame application
- one global outcome
- one promotion decision
- and then a clean stop

That is the current minimal-run standard for the first compatibility launch.
