# COMPATIBILITY_FIRST_PASS_STARTER_CONTEXT_NOTE

## Starter context for the first compatibility-candidate pass

**Date:** 2026-04-07  
**Status:** internal starter-context note

---

## 1. Purpose
This note defines the first concrete starter context for the compatibility first pass.

Its purpose is to answer the next narrowing question:

> **What is the first actual local bridge-adjacent computation context in which Candidate A1 and Candidate B1 should be evaluated under the current compatibility first-pass framework?**

This note is needed because the compatibility block is now methodically very strong, but still requires one final bridge from:
- operational shells
to
- actual first-pass launch context

This is a starter-context note, not a result note.

---

## 2. Why a starter context is needed
The compatibility block already has:

- candidate A1 operationalization
- candidate B1 operationalization
- overlap-only baseline
- later-stabilization boundary
- DLBSR-class
- first-pass comparison frame
- decision logic
- result template
- and input-spec logic

What was still missing was the first actual answer to:

- where exactly does the first pass begin?

Without this, the block remains:
- ready in architecture
but
- not yet ready in local execution context

So the starter context must now be fixed.

---

## 3. Core principle for the starter context
The starter context should obey one central principle:

> **Begin with the narrowest bridge-adjacent local context in which both coherence-side and fragility-side candidate readings are plausible, while later stabilization and readability are not yet forced to dominate.**

A shorter version is:

> **First context: small, bridge-adjacent, early enough for A1 and B1 to speak.**

That is the governing principle.

---

## 4. Recommended starter-context type
The first starter context should be:

> **one local bridge-adjacent neighborhood inside an internal Type-B support-relevant region where local support structure is already nontrivial enough to carry coherence and fragility information, but not yet so late that rigidity-side persistence or readability-side articulation obviously dominate**

A shorter working name is:

> **local bridge-adjacent early-support neighborhood**

This is the recommended first starter-context type.

---

## 5. Why this context should come first
This context is the best first choice because it is the smallest environment that still allows both candidates to be meaningful.

### Why it suits Candidate A1
A1 needs:
- local support relations
- a neighborhood with internal agreement/conflict structure
- more than raw overlap alone

### Why it suits Candidate B1
B1 needs:
- one focal local support unit
- an immediate neighborhood
- and a perturbation context small enough to remain early, but large enough to show local support loss

### Why it does not start too late
A local bridge-adjacent early-support neighborhood can still sit below:
- obvious rigidity-side dominance
- and obvious readability-side dominance

That makes it the correct first starter context.

---

## 6. Concrete working definition
The first starter context should be defined operationally as:

> **one focal local support relation or minimal local support unit plus its immediate bridge-adjacent neighborhood inside an internal Type-B support-relevant region that is already more than weak background but not yet clearly readability-side formed**

This is the current working operational definition.

---

## 7. What the starter context must contain
The chosen starter context must contain:

- one focal local relation or support unit
- one immediate local bridge-adjacent neighborhood
- enough internal local support structure for A1 to be meaningful
- enough local perturbation sensitivity for B1 to be meaningful
- a context still plausibly earlier than strong rigidity-side persistence
- and still plausibly earlier than clearly formed readability-side articulation

These are the required ingredients.

---

## 8. What the starter context must not contain
The starter context should **not** be:

### 8.1 Too weak / too background-like
If local structure is too weak, neither A1 nor B1 is informative.

### 8.2 Too broad
A broad region risks mixing:
- multiple local regimes
- too much late structure
- and too much internal ambiguity

### 8.3 Too readability-side late
If readable articulation is already clearly formed, the compatibility first-pass question is already compromised.

### 8.4 Too stabilization-side late
If rigidity-side persistence already dominates the local reading, Candidate A1 and B1 are too likely to collapse upward.

These exclusions are essential.

---

## 9. Recommended first-pass context rule
A good first-pass context rule is:

> **Choose the narrowest internal Type-B bridge-adjacent local support neighborhood that is already more than weak background and can support both A1 and B1, while still plausibly remaining earlier than strong rigidity-side persistence and clear readability-side articulation.**

This is the preferred full verbal rule.

A shorter version is:

> **Choose the smallest bridge-adjacent local support neighborhood that is already structured but not yet late.**

That is the core rule.

---

## 10. Relationship to overlap-only baseline
The starter context must be rich enough that the overlap-only baseline is not the only thing available.

Why?
Because the first compatibility pass asks whether A1 or B1 adds:
- more than overlap alone

If the local context is too thin, then:
- overlap dominates by default
- and the pass becomes trivial

So the starter context must already be:
- more than bare overlap background

This is important.

---

## 11. Relationship to the later-stabilization boundary
The starter context must also remain early enough that the later-stabilization boundary has a real job to do.

If the context is already too late:
- Candidate A1 may collapse into rigidity-like structure
- Candidate B1 may become only a late dependence readout

So the context should be chosen where:
- the boundary is still a meaningful upper ceiling,
not
- already the main content of the case

That is the correct relation to the boundary.

---

## 12. Relationship to DLBSR-class
The starter context must be late enough that DLBSR-class can still function as a downstream target.

That means the context cannot be so early that:
- no meaningful later local bridge-support response can be discussed at all

So the chosen context should lie in the narrow middle where:
- early candidate signals are meaningful
and
- later support-response remains available as downstream comparison target

This is why the context must be:
- early,
but not too early.
