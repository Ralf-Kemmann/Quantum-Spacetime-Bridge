# COMPATIBILITY_METHODS_INDEX

## Index and workflow overview for the compatibility-methods block

**Date:** 2026-04-06  
**Status:** internal methods index

---

## 1. Purpose
This file is the entry point for the current compatibility-methods block inside the `typ_b_analysis` workspace.

Its purpose is to prevent the compatibility-layer work from dissolving into a flat list of separate notes.

Instead, it provides:
- a thematic map,
- a reading order,
- a workflow order,
- and a compact statement of what this methods block is actually trying to do.

This is an index note, not a result note.

---

## 2. Why this block exists
The compatibility layer is currently the largest open mechanism gap in the `typ_b_analysis` block.

The current project logic already strongly suggests that:
- overlap alone is not enough
- stabilization is too late to be the whole story
- morphology/readability are later still
- therefore an intermediate selection layer is needed

That intermediate layer is currently called:
- the **compatibility layer**

The compatibility-methods block exists to ensure that future work on this layer is:
- operationally disciplined,
- baseline-aware,
- anti-tautological,
- and resistant to premature mechanism claims.

---

## 3. Core idea of the block
The central methodological idea of this block is:

> **A compatibility candidate is only interesting if it can begin to speak before stabilization, outperform generic overlap, and remain distinct from later stabilization-side signals.**

Everything in this block is an unpacking of that principle.

## 7. Recommended practical workflow
The files also define a practical workflow.

### Stage 1 — Define the problem correctly
Use:
- `COMPATIBILITY_LAYER_MINIMAL_PILOT_NOTE.md`

Question:
- what is the smallest useful compatibility question?

---

### Stage 2 — Decide what may count as a candidate
Use:
- `COMPATIBILITY_CANDIDATE_EVALUATION_NOTE.md`
- `COMPATIBILITY_CANDIDATE_PRIORITY_LIST_NOTE.md`

Question:
- which candidates deserve first-pass effort and how should they be judged?

### Stage 3 — Define what they must beat
Use:
- `COMPATIBILITY_BASELINE_COMPARISON_NOTE.md`

Question:
- what lower and upper baselines must every candidate survive?

---

### Stage 4 — Build the first comparison block
Use:
- `COMPATIBILITY_FIRST_TEST_MATRIX_NOTE.md`

Question:
- which two candidates go into the first matrix, against which baselines and which outcome proxy?

### Stage 5 — Execute the first pass conceptually
Use:
- `COMPATIBILITY_FIRST_PASS_WORKFLOW_NOTE.md`

Question:
- in what order should the matrix be processed, and where should the first pass stop?

---

### Stage 6 — Record the result cleanly
Use:
- `COMPATIBILITY_FIRST_PASS_RESULT_TEMPLATE.md`

Question:
- what survived, what was rejected, and what is the single next step?

## 8. Internal philosophy of the block
The compatibility-methods block follows a strict internal philosophy:

- earlier than stabilization
- richer than overlap-only
- resistant to tautology
- narrow first pass
- no rescue logic
- no mechanism-closure language

This philosophy should stay visible whenever a compatibility candidate is discussed.

---

## 9. What this block is trying to prevent
This methods block is specifically designed to prevent:

- overlap being re-labeled as compatibility
- stabilization being mistaken for compatibility
- beautiful analogies replacing early measurable structure
- too many candidates being tested at once
- weak first-pass results being promoted because they are conceptually attractive

That is the block’s negative function.

---

## 10. What success would look like
This methods block has succeeded if, later, a reader can immediately see:

- which candidate families were considered first
- what baselines they had to beat
- why certain candidates were rejected
- why another candidate was only exploratory
- and whether any candidate deserved further refinement

In other words:
success means the compatibility work can be read without guesswork.
